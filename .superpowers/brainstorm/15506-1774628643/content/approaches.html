Three Architectural Approaches

An open sandbox where sound and vision interplay -- solemn, futuristic, tribal

A

Modular Patch Bay (Recommended)

A node-graph interface where audio modules (oscillators, filters, effects, mic input) and visual modules (Hydra sources, transforms, custom shaders) are nodes you wire together. Every connection is bidirectional -- audio FFT feeds visual modulation, visual parameters feed audio synthesis. Users build patches by connecting nodes on a canvas.

Strengths

  • True "endless possibility" -- any signal can route anywhere
  • Visual metaphor matches analog modular synths (tribal/futuristic)
  • Natural save/share: patches are JSON graphs
  • Scales from simple (2 nodes) to complex (dozens)

Risks

  • Most complex to build -- node editor is substantial UI work
  • Steeper learning curve for first-time users
  • Performance: many nodes = many real-time computations
B

Split-Pane Instrument

Full-screen Hydra visual canvas with a collapsible control panel. The panel has sections: Audio Engine (Tone.js synths + mic input), Visual Engine (Hydra parameters + custom shaders), Mappings (which audio signals drive which visual params), and a keyboard/pad grid for playing. Parameters exposed as sliders, knobs, and XY pads.

Strengths

  • Simpler architecture -- no node graph needed
  • Immersive: visuals dominate the screen
  • Faster to build, faster to learn
  • Mapping system still gives deep control

Risks

  • Mapping system is less discoverable than visual wiring
  • "Endless possibility" is more constrained -- predefined parameter slots
  • Adding new module types requires code changes
C

Live Code Arena

A split-screen code editor: left side is Hydra visual code, right side is Tone.js audio code. Both execute live with Ctrl+Enter. A shared "bridge" object lets audio variables flow into visual code and vice versa. The canvas fills the background behind semi-transparent code panels. Presets load as editable code snippets.

Strengths

  • Maximum openness -- code IS the interface, anything is possible
  • Stays true to Hydra's livecoding DNA
  • Simplest architecture -- two code editors + eval
  • Presets are just code, easy to share

Risks

  • Requires coding knowledge -- not accessible to non-coders
  • Less "instrument" feel, more "IDE" feel
  • Security: eval() in browser needs careful sandboxing
